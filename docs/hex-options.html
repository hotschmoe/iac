DESCRIPTION

Traditional hex shapes using / \ _ characters. Each hex is ~7 chars wide × 4 rows tall. Looks unmistakably hex. Most literal translation of a hex grid.

PROS

+ Instantly recognizable as hexagons
+ Room for 3-char content labels inside each hex
+ Missing edges clearly show dead ends

CONS

− Very wide — fits ~8 hexes in 80 cols
− Tall — 4 rows per hex limits vertical density
− Complex rendering logic for partial edges

VERDICT

Beautiful but space-hungry. Best if the map viewport is large and other UI panels are minimal.

DESCRIPTION

Pointy-top hexes using / \ only. Tighter packing — each hex is ~4 chars wide × 2 rows tall. Sacrifices legibility for density.

PROS

+ ~2x more hexes on screen than flat-top
+ Still recognizable as hexagons
+ Simpler rendering (no _ segments)

CONS

− Only 2-3 chars of content per hex
− Feels cramped — hard to read at a glance
− Dead-end edges harder to see clearly

VERDICT

Good density/shape tradeoff. Works if content is minimal icons rather than text labels.

DESCRIPTION

Rectangles in a brick-wall offset pattern. Box-drawing characters for cells, explicit ─ and │ connection lines between them. Hex adjacency is implied by the offset + visible connections.

PROS

+ 5 chars of content per cell — room for labels
+ Box-drawing chars render perfectly everywhere
+ Dead ends are obvious (missing connection lines)
+ Simplest rendering logic by far

CONS

− Doesn't look like hexagons
− Connection lines between rows are complex
− May confuse players expecting a hex map

VERDICT

Most practical. Best information density. Trades hex aesthetics for clarity and simplicity. Strong choice if you label it as a "tactical display" rather than a map.

DESCRIPTION

Each hex is a labeled node [ ]. Connections are drawn as lines between nodes using / \ ─. Dead ends are simply missing lines. Looks like a star chart or network map.

PROS

+ Gorgeous — looks like a real tactical display
+ Dead ends are maximally clear
+ Feels like a constellation chart / star map
+ Connection topology is the visual focus

CONS

− Very low density — ~5 nodes on screen
− Line routing between nodes is hard to compute
− Diagonal lines at arbitrary angles don't work in monospace — limited to / \ diagonals
− Scaling to larger viewports is awkward

VERDICT

Most beautiful option. Best for small, local views (3-4 hex radius). Could pair with a denser option for zoomed-out view.

DESCRIPTION

Uses Unicode diagonal strokes ╱ ╲ for hex edges and ─ for connections. Each hex has room for 3-4 chars of content. Connections between hexes are explicit — missing ─ means dead end. Height: 3 rows per hex. Width: ~8 chars per hex.

PROS

+ Clearly reads as hexagons
+ 3-4 chars of content per cell
+ Dead ends visible (gap where ─ would be)
+ Good density — ~6-7 hex radius in 80×24
+ Unicode diagonals are crisp in modern terminals
+ Brightness levels work beautifully here

CONS

− Requires Unicode support (╱ ╲ chars)
− Slightly more rendering math than brick
− May not render in every terminal font

VERDICT

Best overall balance. Looks hex, shows connections, fits enough on screen, supports the amber brightness-as-info-hierarchy design. This is the recommendation for IAC.

RENDERING NOTES FOR ZIG/ZITHRIL IMPLEMENTATION

All options use axial coordinates (q, r) internally. The rendering function maps (q, r) → (screen_col, screen_row) differently per option, but the game logic is identical regardless of choice.

Brightness mapping: All options benefit from the 4-level amber brightness system. The examples above use: full (player fleet, alerts), bright (enemies, active content), normal (resources, labels), dim (borders, chrome), faint (unexplored, fog of war).

Dead-end rendering: Options 1, 2, 5 show dead ends by omitting edge characters between hexes. Option 3 omits connection lines. Option 4 omits the line entirely. All are clear, but Options 3-5 make dead ends most visually obvious.

Zoom consideration: Options 4 (node graph) and 5 (hybrid) could work as a dual-zoom system — Option 4 for close tactical view during combat, Option 5 for navigation. This would be a zithril feature: pressing a key toggles between zoom levels with different renderers.

Terminal compatibility: Options 1-3 use only basic ASCII and box-drawing characters (universal support). Options 4-5 use Unicode diagonal strokes that require a modern terminal and font. For maximum compatibility, Option 3 is safest. For maximum beauty, Option 5.
